# WorldFish 宪章

> 宪章的价值在于被记住和被执行。本文档只保留最高约束；细节、模板和检查清单应放入独立 Spec 或流程文档，不得把宪章扩写成没人阅读的长文档。

## 0. 一句话原则

**WorldFish 主仓库是本地世界观工作台，不是云平台仓库。所有取舍遵守：扩展性 > 本地可用 > 创作者体验 > 云端可用。**

## 1. 仓库定位

- `WorldFish`：本地开源主仓库，负责世界观管理、知识组织、RAG / Agent、推演、报告、本地存储、本地模块、导入导出、局域网联机与自托管协作。
- `WorldFish-Cloud`：云平台仓库，负责官方云账号、团队空间、公网协作、云同步、云存储、云 AI 编排、计费、市场、运营后台等云服务能力。
- `WorldFish-Shared`：未来可选共享协议仓库，适合放 Schema、导入导出协议、纯函数和跨仓库复用逻辑。

主仓库可以设计接口、Schema、Adapter 和 Capability；不得实现云平台业务。

## 2. 强制优先级

1. **扩展性优先**：模块、数据协议、AI 编排和未来云平台接入空间不得被短期实现锁死。允许为了长期扩展性牺牲短期开发速度。
2. **本地可用**：clone 仓库后，按文档安装依赖并配置必要参数，应能运行本地工作台。允许下载依赖、调用用户自行配置的 LLM / Embedding 服务；不得要求 WorldFish 官方云账号或云服务才能启动基础能力。
3. **创作者体验**：功能应操作简单、表达自由、数据清晰、AI 辅助强，并保留创作者对世界观数据和 AI 输出的控制权。
4. **云端可用**：当前主仓库不实现云平台能力，但不能阻碍 `WorldFish-Cloud` 未来推动接口、Schema、模块边界和协议演进。

低优先级目标不得牺牲高优先级目标。

## 3. 云边界禁令

主仓库不得实现：

- 官方云账号、登录、团队、组织、邀请、商业权限。
- 官方公网多租户协作、云端 WebSocket 网关、云端协作房间。
- 云端项目同步、云端存档、云端版本历史。
- 云端对象存储、CDN、资源托管、素材库。
- 云端 RAG、云端向量库、云端 AI 编排、官方模型额度。
- 计费、订阅、套餐、额度、支付、市场、运营后台、审计、风控、合规后台。
- 云平台私有密钥、私有环境变量、闭源部署脚本或闭源服务实现。

判断标准：如果实现需要 WorldFish 官方云服务承担账号、数据、资源、权限、计费、运营或公网托管责任，它就属于 `WorldFish-Cloud`。

禁止用中性命名规避边界。例如把云账号称为“统一身份”、把云同步称为“远程工作区”、把云存储称为“资源服务”、把计费称为“能力等级”、把公网托管协作称为“增强联机”，都仍然算云平台实现。

局域网联机、自托管协作、本地假实现可以留在主仓库，但必须不依赖 WorldFish 官方云账号、云服务、计费、订阅或公网托管。

## 4. Spec 最小要求

非平凡变更必须先写 Spec，再实现。包括：新增或重做功能、跨模块改动、核心数据 / Schema / API 变化、AI / RAG / Agent 编排变化、协作 / 同步 / 权限变化、新依赖或外部服务、任何可能影响仓库边界或本地可用的变更。

小型 bug 修复、文案、注释、样式、测试补充、构建脚本修正、文档错字，可以不单独写 Spec，但仍必须遵守本宪章。

合格 Spec 只强制包含 6 项：

1. **问题与目标**：解决什么，为什么现在做。
2. **非目标与边界**：本次不做什么，哪些云平台能力不得进入主仓库。
3. **方案概述**：前端、后端、数据、API、AI / RAG / Agent、模块边界如何变化。
4. **扩展性与本地可用影响**：是否锁死模块、协议、AI 编排、云扩展；是否影响 clone 后本地运行。
5. **验收标准**：用可判定条件说明什么算完成，不得只写“体验更好”“基本可用”。
6. **测试、迁移与风险**：如何验证，旧数据如何兼容，失败如何回滚，还有什么未决风险。

Spec 缺失、不完整或违反宪章时，AI 协作者必须停止实现，先修正 Spec。

## 5. 实现与验收

实现前必须判断能力归属：`WorldFish` 本地能力、`WorldFish-Cloud` 云平台能力、未来 `WorldFish-Shared` 共享协议，或主仓库中的抽象接口。

实现必须避免：

- 把核心业务规则硬编码进单个 Vue 页面、Flask 路由或临时脚本。
- 把核心数据写成不可解释、不可迁移、不可导出的临时结构。
- 把 AI 流程绑定到单一模型供应商，或写死 API Key。
- 在无确认、无追踪、无回滚路径时让 AI 覆盖创作者核心数据。
- 引入会污染仓库边界的大型依赖或云服务 SDK。

完成前必须确认：符合 Spec、未违反云边界、本地路径未被破坏、扩展性未被短期实现锁死。涉及云边界时，应运行或说明为何未运行：

```bash
npm run check:oss-boundary
```

未测试项必须说明原因、风险和后续验证方式。

## 6. 违反宪章的处理

发现需求、Spec、设计或实现违反本宪章时，必须立即停止相关实现。

处理顺序：

1. 指出违反的条款。
2. 修正 Spec 或设计。
3. 删除、迁移或重写违规实现。
4. 必要时迁移到 `WorldFish-Cloud`，或仅在主仓库保留接口 / Schema / Adapter。
5. 重新验收。

不得以“临时方案”“先这样”“后面再改”“已经写完”为理由绕过宪章。

## 7. 篇幅约束

本宪章必须保持短小。新增条款应优先合并、替换或删除旧条款，而不是持续追加。若某内容需要示例、模板、流程或长解释，应移出本文档，放入独立 Spec 模板或开发流程文档。
