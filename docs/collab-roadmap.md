# WorldFish 联机协作路线图

本文档描述 WorldFish 联机 / 协作能力的演进路线。每个阶段都建立在前一阶段之上，
保持「单机即单人房间」的核心理念，逐步走向多人在线协作平台。

## 当前已完成（生产级底座）

- 持久化：SQLite（WAL 模式），房间 / 成员 / 事件均走索引查询
- 事件序列：数据库事务内分配房间内递增 `seq`，并发安全且唯一
- 身份与权限：`owner` / `editor` / `viewer` 三级角色，写操作校验成员与角色
- 输入校验：名称、描述、消息、payload 大小与 ID 字符白名单，统一错误码
- 乐观并发：世界观保存携带 `base_seq`，检测到他人更新返回 409
- 增量同步：`list_events` 返回 `cursor` / `has_more`，`/sync` 长轮询端点降低延迟
- 可观测性：协作操作结构化日志，`/health` 健康检查
- 兼容迁移：检测旧 JSON 数据时一次性导入 SQLite

---

## 阶段 A — 真实多用户身份

目标：从「单一 local_user」走向真实账号体系。

- 账号注册 / 登录（用户名 + 密码，bcrypt/argon2 哈希）
- 会话令牌（短期 access + 刷新机制），替代当前 `X-WorldFish-User` 明文头
- 可选 OAuth / 第三方登录
- `local_user` 平滑迁移为首个真实账号
- 工作区与房间归属到真实用户

风险点：迁移期需保持本地单机零配置可用，登录态缺失时回退到本地房间。

---

## 阶段 B — 实时传输

目标：用推送替代轮询，降低延迟与服务端压力。

- WebSocket 网关（房间维度订阅）
- 服务端事件广播：append_event 后即时推送给房间在线成员
- 在线状态实时化（连接建立 / 断开即更新 presence）
- 前端从 `/sync` 长轮询切换到 WebSocket，长轮询作为降级回退
- 心跳与重连策略

依赖：阶段 A 的会话令牌用于 WebSocket 鉴权。

---

## 阶段 C — 协同编辑

目标：从「房间事件 + 整体保存」走向字段级并发编辑。

- 第一步：字段级编辑锁（谁在编辑哪个实体 / 设定）
- 第二步：操作变换（OT）或 CRDT，支持同一世界观多人同时编辑
- 冲突合并策略与可视化（谁改了什么）
- 编辑历史与逐操作回放

风险点：复杂度高，需在阶段 B 的实时通道稳定后再做。

---

## 阶段 D — 横向扩展

目标：从单机 SQLite 走向多 worker / 多实例部署。

- 存储：SQLite → PostgreSQL（保持 store 接口不变，替换实现）
- 事件总线：Redis Pub/Sub 或 NATS，实现跨进程 / 跨实例事件广播
- 多 worker 下的 WebSocket 网关粘性会话或共享订阅
- 读写分离与连接池

设计前提：当前 `CollabStore` 已抽象数据访问，迁移时只替换实现层。

---

## 阶段 E — 治理与商业化

目标：满足开源 / 闭源分层与运营需求。

- 审计日志（谁在何时对哪个房间做了什么）
- 配额与限流（按用户 / 房间 / 工作区）
- 完整权限矩阵（自定义角色、细粒度权限）
- 房间归档、导出与回放
- 开源核心（单机 + 基础协作）与闭源联机增强（多租户、计费、运营后台）分层

---

## 设计原则（贯穿所有阶段）

1. 单机优先：任何阶段都保证本地单人零配置可用
2. 接口稳定：存储层、事件层、权限层接口先于实现稳定
3. 渐进增强：实时 / 协同编辑作为增强，缺失时回退到可用基线
4. 事件溯源：所有协作动作沉淀为 append-only 事件，便于同步、审计与回放
